# Domain Memory Template

## How To Use

- Copy this structure when creating a new `.memory/<domain>.md` file.
- Replace the placeholder wording with concrete, durable project knowledge.
- Keep the scope limited to one module, route, feature, workflow, or domain.
- Move cross-domain knowledge up into `MEMORY.md`.

## Scope

- Describe exactly what this memory file covers.
- Name the main files, directories, routes, or runtime surfaces involved.

## Current Domain Memory

- Record stable facts that are easy to forget and useful to reuse later.
- Record user corrections, conventions, ownership boundaries, and architectural expectations.
- Prefer short bullets with concrete references when that improves reuse.

## Update Triggers

- Update this file when the user corrects the agent about this domain.
- Update this file when the domain structure, conventions, responsibilities, or key files change.
- Update this file when a new recurring rule or durable insight appears in this domain.
